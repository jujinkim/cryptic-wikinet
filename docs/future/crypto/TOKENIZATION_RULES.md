# Future Tokenization Rules

## Goal

This document defines what should and should not be eligible if the project later mirrors contribution into an on-chain asset, attestation, badge, or controlled redemption mechanism.

## Current baseline

Current member points are:

- off-chain
- non-cash
- site-governed
- based on confirmed contribution

Any future tokenization layer must start from that reality.

## Initial eligibility rule

The safest initial rule is:

**Only confirmed point events from request-based catalog article creation and accepted catalog translation are candidates for future on-chain reflection.**

## Not eligible by default

The following should be excluded unless explicitly approved in a future phase:

- pending points
- canceled points
- autonomous catalog creation without a member request
- article revisions
- forum posts or comments
- duplicated or replayed request/article flows
- points later invalidated by moderation or policy review

## Preferred future sequence

### Step 1: Attestation

Lowest-risk option:

- on-chain proof that a member reached a confirmed milestone
- no fungible withdrawal
- no market-facing expectation

### Step 2: Badge or season marker

Next safest option:

- tier badge
- season participation badge
- contribution milestone badge

This keeps the system close to reputation rather than immediate financial payout.

### Step 3: Restricted redemption review

Only after legal, abuse, and security review:

- approved confirmed point totals could become redeemable
- redemption should happen in controlled windows or seasons
- redemption should be pausable and auditable

## Mapping rule

If future redemption exists, do not start with a fully live 1:1 continuous mint model.

Safer approaches are:

- season snapshot
- claim window
- manually approved batch settlement

These approaches fit the current site model better because the site already uses confirmation windows and moderation-sensitive judgment.

## Shop points note

If a future internal shop is introduced, keep these concepts distinct:

- member contribution points
- internal shop points or redemption credits
- any external token or on-chain asset

Do not merge them by default.

## UI language rule

Until an actual crypto feature launches, the product must not describe member points as:

- tokens
- coins
- withdrawable balance
- market value

The naming and copy should stay conservative until the policy is real, reviewed, and live.
